Usage data tells Doow what is being consumed. Choose the source based on where
the usage data already exists, who controls the system producing it, and
whether Doow needs to collect activity directly from a user's browser or
desktop.

## Recommended default

If you need to understand employee activity across browser-based SaaS apps or
native desktop apps, start with Doow Agent. Browser Agent and Desktop Agent are
the same Doow usage-collection product across different device surfaces.

If the provider has a direct usage API and that provider is the source of
record, start with
[Direct provider APIs](/integrations/usage/direct-provider-apis). This is the
lowest-effort path for AI, API, and platform usage because Doow pulls usage
directly from the provider of record.

Use the [Instrumentation SDK](/integrations/usage/instrumentation-sdk) when
you need custom metrics that vendors do not expose. Use
[Observability integrations](/integrations/usage/observability) when your
metrics already exist in a monitoring platform.

## Compare the paths

| Path | Best for | Examples |
|---|---|---|
| Doow Agent | Browser and desktop app activity collected from user devices | Browser Agent, Desktop Agent |
| Direct provider APIs | Provider-owned usage data | OpenAI, Anthropic, Groq, Fireworks AI, OpenRouter, Replicate, Authress |
| Cloud platforms | Usage exposed through cloud monitoring or billing | AWS Bedrock, Azure OpenAI, Google Vertex AI |
| Instrumentation SDK | Product usage you emit yourself | Node.js SDK, serverless wrappers, sidecar, CLI |
| Observability | Metrics already collected in monitoring tools | Datadog, New Relic, AppSignal, Sentry |
| OTLP push | Metrics already flowing through OpenTelemetry | OpenTelemetry Collector |

## What success looks like

After setup, Doow should show usage records for the source you connected. For
Doow Agent, that means browser or desktop activity appears for the expected
apps and users. For provider, cloud, SDK, observability, or OTLP paths, that
means usage appears for the expected provider, project, account, model,
deployment, service, or metric.

If the connection succeeds but no usage appears, check that the source is
enabled for the right users, workspace, organization, project, account, region,
or metric stream.
